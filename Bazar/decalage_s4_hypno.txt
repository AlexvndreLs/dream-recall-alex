Vérification alignement 30s des hypnogrammes (sleep_raw_data/hypnograms)
==========================================================================

Contexte
--------
Hypnogrammes annotés par seconde (1 valeur/ligne, txt). Hypothèse de travail :
le scoring source est par epoch de 30s, chaque seconde de l'epoch portant le
même label. Pour que le découpage `range(0, len(secs), 30)` (split_cycles /
merge_S3_S4 dans utils.py) soit valide, toutes les transitions de stage
doivent tomber sur des indices multiples de 30, à partir de l'indice 0.

Méthode
-------
Script HypnogramAligner :
- charge le fichier (1 int/ligne)
- repère les indices de transition (np.diff != 0)
- calcule offset = indice_transition % 30
- calcule aussi n_samples % 30 (longueur de la dernière epoch incomplète)

Résultats
---------
- Préfixe hyp_per_s* (38 fichiers) : 37/38 ont offset_distribution = {0: N},
  donc transitions parfaitement alignées sur des epochs de 30s dès l'indice 0.
- Exception : hyp_per_s4.txt -> offset_distribution = {29: 119},
  n_samples = 21609. Décalage constant de -1 (29 = 30-1).
- Préfixe hyp_jbe_s* (18 fichiers, second annotateur, sous-ensemble) :
  18/18 ont offset 0, y compris hyp_jbe_s4.txt (n_samples = 21610).

Diagnostic
----------
hyp_per_s4 a très probablement perdu le tout premier échantillon (1 seconde)
par rapport à hyp_jbe_s4 (même sujet, même nuit, n_samples = 21610 vs 21609,
différence = 1, et jbe_s4 est aligné sur 0). Pas un décalage global du
dataset : cas isolé sur s4.

length_remainder (n_samples % 30) varie selon les sujets (0 à 29) : normal,
correspond à la dernière epoch incomplète en fin de nuit, déjà ignorée par
le découpage par paquets de 30 (range s'arrête sans yield le reste).

Correction retenue
-------------------
Pour s4 uniquement : préfixer le hypnogramme (ou la série équivalente côté
BIDS) d'une valeur égale à hypno[0], pour repasser n_samples=21609 -> 21610
et offset 29 -> 0, cohérent avec hyp_jbe_s4.

Conclusion
----------
Le découpage en epochs de 30s à partir de l'indice 0 est valide pour
l'ensemble du dataset sous réserve de cette correction ponctuelle sur s4.
Pas de problème d'alignement systémique.

Localisation de l'échantillon manquant (début vs fin) - hyp_per_s4
--------------------------------------------------------------------
Comparaison directe avec hyp_jbe_s4 (même sujet, second annotateur) :
- shift 0  : 4011 mismatches / 21609
- shift +1 (a[1:] vs b)   : 4052 mismatches / 21608
- shift -1 (a vs b[1:])   : 3970 mismatches / 21609
=> ~18-19% de désaccord quel que soit le shift : désaccord inter-annotateur
classique (kappa ~0.7-0.8 typique en scoring du sommeil), non discriminant
pour localiser l'échantillon manquant.

Preuve indépendante via la structure interne de hyp_per_s4 :
offset_distribution = {29: 119} sur les 119 transitions, de façon
constante. En découpant en blocs [0:29], [29:59], [59:89], ... (1er bloc
de 29 échantillons, puis blocs de 30), tous les blocs sont homogènes.
=> Le premier epoch est tronqué à 29s au lieu de 30s : l'échantillon
manquant est en DEBUT de fichier, pas en fin.

Conclusion finale : padding d'1 échantillon (= hypno[0]) en tête de
hyp_per_s4 est la correction correcte et suffisante.

À vérifier ensuite
-------------------
- Confirmer que hyp_per_s4 et hyp_jbe_s4 sont identiques (à 1 échantillon
  près) sur le reste de la série, pour écarter une vraie perte de données
  en début de nuit (et pas juste un artefact de fichier).
- Si correction appliquée côté BIDS : vérifier le format de stockage de
  l'hypnogramme (events.tsv en epochs vs série continue) avant de
  décider entre "rien à faire" et "préfixer + décaler onset de -1s".
